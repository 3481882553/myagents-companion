| 层级 | 选型 | 理由 |

| **SSE 客户端** | `react-native-sse` | EventSource polyfill，兼容现有 SSE 协议 |
| **状态管理** | Zustand + React Query | 轻量、支持离线缓存 |
| **样式** | NativeWind (Tailwind for RN) | 复用桌面端设计令牌 |
| **本地存储** | MMKV + SQLite | MMKV 配置，SQLite 消息缓存 |
| **深度链接** | React Navigation Linking | 开箱即用 |
| **认证** | `jose`（JWT 库） | Node.js 生态成熟，支持 JWE/JWS |
| **小助理** | 标准 Chat 会话 + 预置 CLAUDE.md | 本质是普通会话，通过 Sidecar Admin API 管理应用 |

### 2.1 架构分层

| 层 | 职责 | 技术 |
|---|---|---|
| **渲染层** | 将消息数据渲染为 UI | RN 原生组件 + WebView（Mermaid/KaTeX/Widget） |
| **通信层** | 与桌面端 Sidecar 通信 | HTTP 客户端 + SSE 客户端 + 重连状态机 |
| **原生能力层** | 调用手机原生功能 | 深度链接、二维码扫描、剪贴板 |
| **状态层** | 管理应用状态和缓存 | Zustand store + SQLite 缓存 |

### 2.2 关键设计决策

**Q: 为什么不用 WebSocket？**
A: 复用现有 SSE 协议，零改造成本。SSE 天然支持服务端推送，单向流足够（移动端发消息走 HTTP POST）。

**Q: 为什么 WebView 渲染 Mermaid/KaTeX？**
A: mermaid 库内部用 `document.createElement` 创建 SVG，KaTeX 依赖 DOM。RN 没有 DOM，只能用 WebView 桥接。桌面端源码中 `MermaidDiagram.tsx` 使用 `MutationObserver` 监听主题变化，`PdfViewer.tsx` 使用 `document.createElement('canvas')` + `IntersectionObserver`，这些都无法移植到 RN 原生。

**Q: 为什么离线缓存用 SQLite 而不是 AsyncStorage？**
A: 消息量大（1000 条 × 多会话），SQLite 支持结构化查询和分页，性能远优于 KV 存储。

## 三、通信协议设计

### 3.1 复用现有协议

移动端**复用桌面端的 HTTP + SSE 协议**，不引入新协议。所有 API 端点与桌面端并行存在。

### 3.2 API 端点规划

| 端点 | 方法 | 用途 | 说明 |
|------|------|------|------|
| `/api/mobile/pair` | POST | 配对码验证 → 返回 JWT | 首次连接使用 |
| `/api/mobile/sessions` | GET | 列出可用会话 | 支持分页 |
| `/api/mobile/session/:id` | GET | 获取会话完整消息 | 支持分页加载 |
| `/api/mobile/session/:id/stream` | GET (SSE) | 实时流式推送 | 复用现有 SSE 协议 |
| `/api/mobile/session/:id/send` | POST | 发送消息 | 文本消息 |
| `/api/mobile/approve/:request_id` | POST | 工具权限审批 | 允许/拒绝 |
| `/api/mobile/logs` | GET | 日志查询 | 关键词搜索 + 时间范围，小助理诊断用 |

### 3.3 SSE 事件处理策略

> ⚠️ **源码验证**：`src/server/sse.ts` 的 `SSE_EVENT_PRIORITIES` 定义了 52 个事件，`SseConnection.ts` 的 `JSON_EVENTS` 定义了 49 个，并集 53 个。以下分类以 `SSE_EVENT_PRIORITIES` 为准。

移动端需处理的 SSE 事件分为三级：

| 级别 | 策略 | 数量 | 事件 |
|------|------|------|------|
| **critical** | 必须处理 | 41 | `chat:message-complete`, `chat:message-error`, `chat:message-stopped`, `chat:message-replay`, `chat:message-sdk-uuid`, `chat:system-init`, `chat:system-status`, `chat:status`, `chat:init`, `chat:tool-use-start`, `chat:tool-result-start`, `chat:tool-result-complete`, `chat:tool-attachment-update`, `chat:server-tool-use-start`, `chat:subagent-tool-use`, `chat:subagent-tool-result-start`, `chat:subagent-tool-result-complete`, `chat:thinking-start`, `chat:content-block-stop`, `chat:agent-error`, `chat:api-retry`, `chat:permission-mode-changed`, `chat:session-title-changed`, `chat:task-started`, `chat:task-notification`, `chat:attachments-filtered`, `chat:attachments-fallback`, `permission:request`, `ask-user-question:request`, `ask-user-question:expired`, `enter-plan-mode:request`, `enter-plan-mode:expired`, `exit-plan-mode:request`, `exit-plan-mode:expired`, `cron:task-exit-requested`, `mcp:oauth-expired`, `config:changed`, `plugin:install-progress`, `plugins:changed`, `queue:added`, `queue:started`, `queue:cancelled` |
| **coalescible** | 可合并（40ms 窗口） | 7 | `chat:message-chunk`（⚠️ 源码中是 coalescible 非 critical）, `chat:thinking-chunk`, `chat:tool-input-delta`, `chat:tool-result-delta`, `chat:subagent-tool-input-delta`, `chat:subagent-tool-result-delta`, `chat:context-usage` |
| **droppable** | 直接忽略 | 4 | `chat:log`, `chat:logs`, `chat:debug-message`, `chat:runtime-diagnostics` |

共 52 个已注册事件（`SSE_EVENT_PRIORITIES`），另有 1 个仅在 `JSON_EVENTS` 中（`chat:message-sdk-uuid` 已在 critical 中）。详见详细设计文档。

## 四、安全设计

### 4.1 分阶段策略

| 阶段 | 传输 | 认证 | 适用场景 |
|------|------|------|---------|
| **Phase 1（MVP）** | HTTP | 配对码 → JWT Token | 局域网内 |
| **Phase 2（增强）** | HTTPS + 证书钉扎 | 配对码 → JWT Token | 外网场景 |

### 4.2 认证流程

```
1. 桌面端开启「允许移动端连接」
2. 生成 6 位配对码 + 二维码（包含 host + 配对码）
3. 手机扫码或手动输入配对码
4. POST /api/mobile/pair { code: "123456" }
5. 验证通过 → 返回 JWT Token（7 天有效）
6. 后续请求携带 Authorization: Bearer <token>
7. Token 过期前 24 小时自动续期
```

### 4.3 安全措施汇总

| 层面 | 措施 |
|------|------|
| 认证 | 6 位配对码 → JWT Token，`crypto.randomInt` 生成 |
| 请求 | 所有请求携带 `Authorization: Bearer <token>` |
| 存储 | Token 存储在 Keychain/Keystore 加密存储 |
| 输入 | 用户消息长度限制 4000 字符 |
| 渲染 | rehype-sanitize XSS 防护 |
| 设备管理 | 桌面端可查看/撤销已配对设备 |

### 4.4 为什么不用 CORS

React Native 的 HTTP 请求（fetch/axios）不经过浏览器，没有 Origin 概念。CORS 是浏览器安全机制，对移动端原生 HTTP 请求无效。移动端安全通过 JWT Token 验证实现。

## 六、设计体系对齐

### 6.1 设计原则

移动端与桌面端共享**同一套设计语言**——「暖色纸质」（Warm Paper / Warm Night）。核心原则：

- **色温一致**：暖色调为主，不使用冷色系（纯蓝/纯紫）作为主色
- **质感一致**：保持桌面端的纸质质感（点阵纹理可简化，但色温不变）
- **交互一致**：相同操作在两端的视觉反馈一致（按钮按下、链接颜色等）

| SC-04 | IM 消息深度链接 | 低 | `src-tauri/src/im/mod.rs` 的 `finalize_block` |
| SC-05 | 日志查询 API | 低 | 新增 `/api/mobile/logs` 端点，小助理诊断用 |

### 7.2 SC-01 绑定地址可配置

**当前状态**：`src/server/index.ts` 第 1816 行硬编码 `hostname: '127.0.0.1'`，代码注释明确说明 "Explicit 127.0.0.1 for Rust proxy compatibility (IPv4)"。

**⚠️ 关键约束**：当前安全模型完全依赖 127.0.0.1 绑定（环回信任模型），Sidecar 无任何认证中间件。直接改为 0.0.0.0 会：
1. 破坏 Rust 代理层 `local_http.rs` 的 `no_proxy()` 逻辑（依赖环回地址判断）
2. 暴露所有 Admin API 到局域网（当前无认证）

**改造方案**（分两步）：

**Step 1：新增独立的移动端 HTTP 服务**（不修改现有 Sidecar 绑定）
- 在 Sidecar 中新增第二个 HTTP 服务，绑定 `0.0.0.0:mobilePort`
- 仅暴露移动端需要的端点（会话读取、消息发送、权限审批）
- 所有请求经过 JWT 认证中间件
- 现有 127.0.0.1 服务保持不变，Rust 代理层零改动

**Step 2：Rust 代理层联动**（最小改动）
- `src-tauri/src/sidecar.rs`：Sidecar 启动时传入 `--mobile-port` 参数
- `src-tauri/src/sse_proxy.rs`：`request_target_is_loopback()` 不需要改动（移动端直连 Sidecar，不经过 Rust 代理）
- `src-tauri/src/local_http.rs`：不需要改动

**改动量修正**：SC-01 从"简单配置"提升为"中等复杂"，需新增独立 HTTP 服务 + 端口管理

### 7.3 SC-02 配对码 + JWT

**⚠️ 关键约束**：当前 Sidecar 无任何认证中间件，安全模型完全依赖 127.0.0.1 环回绑定。代码注释："the sidecar is bound to 127.0.0.1 and trusts everything on loopback already"。

**改造方案**：在移动端 HTTP 服务（SC-01 Step 1）上实现独立的认证层，不影响现有 Sidecar：

- 配对码：`crypto.randomInt(100000, 999999)` 生成 6 位数字
- JWT：使用 `jose` 库签发，有效期 7 天
- 认证中间件：仅作用于移动端 HTTP 服务的路由，不修改现有 127.0.0.1 服务的访问控制
- 续期：Token 过期前 24 小时自动续期
- SSE 长连接：连接建立时验证 Token，后续心跳检测连接存活（Token 过期则断开，客户端重新认证）

**安全模型变更说明**：
- 现有模型：环回信任（127.0.0.1 = 可信）
- 新增模型：移动端 = 独立 HTTP 服务 + JWT 认证（与环回信任并行，互不影响）
- 不是"修改现有认证"，而是"新增独立认证层"

### 7.4 SC-03 移动端 API

> ⚠️ **源码验证**：现有端点使用手动 pathname 匹配（`new URL(req.url).pathname`），不是 Hono 路由 API。端点命名风格为单数（`/api/session/messages`），不是复数。新增端点应遵循现有风格。

新增端点（在移动端独立 HTTP 服务上，遵循现有命名风格）：

| 端点 | 方法 | 用途 | 说明 |
|------|------|------|------|
| `/api/session/list` | GET | 列出可用会话 | 复用现有 `sessionStore.listSessions` |
| `/api/session/messages` | GET | 获取会话消息 | 复用现有消息查询逻辑 |
| `/api/session/stream` | GET (SSE) | 流式推送 | 复用现有 SSE 逻辑 |
| `/api/session/send` | POST | 发送消息 | 复用现有消息入队逻辑 |
| `/api/permission/respond` | POST | 权限审批 | 复用现有权限处理逻辑 |
| `/api/pair` | POST | 配对码验证 | 新增，返回 JWT |

**不新增 `/api/logs/export`**：现有端点已存在，移动端直接复用。

预计改动量：500-800 行 TypeScript（移动端 HTTP 服务 + 路由）。

## 九、Widget 沙箱方案

### 9.1 安全模型

桌面端 Widget 使用 `<iframe sandbox="allow-scripts">`（无 `allow-same-origin`）+ 严格 CSP + `postMessage` 通信。

移动端用 `react-native-webview` 实现等效安全边界：

| 桌面端 | 移动端 |
|--------|--------|
| `<iframe sandbox="allow-scripts">` | 独立 WebView 实例 |
| CSP `default-src 'none'` | WebView `javaScriptEnabled` + 内容过滤 |
| `postMessage` 通信 | `onMessage` / `injectJavaScript` |
| `sanitizeForStreaming()` 剥离 script/on* | 同样逻辑，在注入前执行 |

### 9.2 降级策略

| 条件 | 降级方式 |
|------|---------|
| WebView 初始化失败 | 显示"渲染组件加载失败"，提供重试 |
| 渲染超时（5s） | 降级为代码文本 |
| WebView 内存溢出 | 关闭实例，降级为代码文本 |
| 连续 3 次失败 | 该会话后续 Widget 都降级 |

## 十、内置小助理方案

### 10.1 架构定位

小助理是 MyAgents 的内置 AI 助手，帮助用户配置应用、排查问题、提交反馈。桌面端已有此功能，移动端需要对等支持。

**核心发现**：小助理本质上是一个**普通聊天会话**，只是带了特殊的系统提示词（CLAUDE.md）和专属 skill。不是独立模块。

```
首页入口（卡片/浮动按钮）
  │  点击
  ▼
创建/恢复小助理会话
  │  workspace = ~/.myagents/
  │  system prompt = bundled CLAUDE.md
  │  skills = support skill
  ▼
标准 Chat 界面
  │  用户提问 → Sidecar 处理 → SSE 返回
  ▼
小助理通过 Sidecar Admin API 执行操作
  ├── /api/provider/*  → 管理模型供应商
  ├── /api/mcp/*       → 管理 MCP 服务器
  ├── /api/admin/cron/* → 管理定时任务
  └── /health/*        → 检查系统状态
```

### 10.2 与桌面端的差异

| 方面 | 桌面端 | 移动端 |
|------|--------|--------|
| **入口** | 标题栏按钮 → 弹窗 → 全屏遮罩 | 首页卡片 / 浮动按钮 |
| **管理能力** | `myagents` CLI（直接执行） | Sidecar Admin API（HTTP 调用） |
| **日志诊断** | `grep` 日志文件 | 新增 `/api/mobile/logs` 端点 |
| **Issue 提交** | `gh issue create` | GitHub API 或分享链接 |
| **会话持久化** | 标准 Session 持久化 | 同（SQLite 缓存） |

### 10.3 打包内容

| 文件 | 来源 | 用途 |
|------|------|------|
| `CLAUDE.md`（597 行） | `bundled-agents/myagents_helper/` | 系统提示词：产品知识、诊断流程、CLI 参考 |
| `support/SKILL.md`（183 行） | `bundled-agents/myagents_helper/.claude/skills/` | 诊断 skill：诊断 → 分类 → 行动 |

首次连接 Sidecar 时，移动端将这些文件同步到 Sidecar 的 `~/.myagents/` 目录。

### 10.4 开发阶段价值

小助理在 MVP 阶段末期即可投入使用，作为移动端开发的实时调试助手：
- 开发中遇到连接/渲染问题 → 直接在手机上问小助理
- 小助理通过 Admin API 诊断 Sidecar 状态
- 比翻日志更快定位问题

## 十二、推送通知方案

### 12.1 触发场景

| 场景 | 触发条件 | 通知内容 |
|------|---------|---------|
| AI 回复完成 | SSE 收到 `chat:message-complete` | "AI 已回复：{摘要前 50 字}" |
| 权限审批请求 | SSE 收到 `permission:request` | "AI 请求使用 {工具名}，请审批" |
| 任务完成 | SSE 收到 `chat:task-notification` | "任务 {任务名} 已完成" |

### 12.2 技术方案

| 平台 | 推送服务 | 方案 |
|------|---------|------|
| Android | 本地通知 | `react-native-push-notification` |
| iOS | 本地通知 | `@react-native-community/push-notification-ios` |

**通知方案说明**：

移动端与 Sidecar 的连接是局域网 SSE，没有中心化服务器做远程推送。因此采用**本地通知**方案：

- **前台时**：SSE 事件直接驱动 UI 更新（不需要通知）
- **后台时**：SSE 连接断开，无法接收事件 → 使用**本地通知**兜底
- **通知触发时机**：App 进入后台前，将前台收到的关键事件缓存为本地通知

**FCM/APNs 的定位**（依赖库中保留 `@react-native-firebase/messaging`）：
- 当前仅用于获取设备 Token（为未来远程推送做预留），**不做实际远程推送**
- 如果未来有中心化服务器，可切换为 FCM/APNs 远程推送

### 12.3 权限管理

- 首次启动时请求通知权限
- 设置页面提供通知开关
- 可按场景独立控制（AI 回复 / 权限审批 / 任务完成）
