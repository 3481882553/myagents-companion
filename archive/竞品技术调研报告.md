### 4.1 Markdown 渲染管线


### 4.2 流式渲染的 Markdown 前缀固化策略（核心技巧）

```
第 1 次 flush：[已渲染 0 字符] → 全部 markdown 渲染
第 2 次 flush：[已渲染 120 字符] → 前缀 120 字固化为 Markdown，新 30 字纯文本追加
第 3 次 flush：[已渲染 150 字符] → 前缀 150 字固化为 Markdown，新 20 字纯文本追加
...
```

**好处**：
- 避免每次收到新字符都触发完整的 markdown AST 重建
- markdown 前缀保持布局稳定（不会因为新字符到来导致整个块重新排版）
- 新尾部用纯文本 + Opacity 渐入，视觉上流畅

## 六、关键发现与启示

### 6.1 流式渲染是最大差距

当前 Companion 用 `setTimeout(3000ms)` 轮询，而三个参照项目都用 SSE 实时流式。最严重的不是"慢 3 秒"，而是：
- 长消息的中间状态完全不可见
- 工具调用执行过程看不到
- 思考过程无法实时展示

**解决方案已就绪**：SseClient（52 事件类型）+ EventCoalescer（40ms 批处理）代码已完成，只需接入 ChatScreen。

### 6.2 OpenOmniBot 的前缀固化策略最值得借

这是我们当前的实现路径：

```
当前（如果接入 SSE）：
  chat:message-chunk → 拼接完整文本 → react-native-markdown-display 全量重渲染
  问题：每次 chunk 都触发完整 markdown AST 重建，长消息会越来越卡

改进后（参照 OpenOmniBot StreamingText）：
  已完成的前缀（>200 字符且稳定）→ 固化为 markdown widget，不重渲染
  新来的尾部（<200 字符）→ 纯文本追加，Opacity 渐入
  每 10 个 chunk flush 一次 → 重新 markdown 渲染全文
```

### 6.3 Operit 的 KMP 流式解析对 RN 不适用

Operit 的优势来自于：
1. C++ 原生代码加速
2. Compose Canvas 直接绘制（跳过 DOM/RN View 层级）
3. 自研字符级状态机

这些在 React Native 架构中无法直接复制。RN 的渲染是 JS → Native Bridge → Yoga 布局 → Native View，层级开销远大于 Compose Canvas。RN 的最佳策略是**减少重渲染次数**（前缀固化），而不是**加速每次渲染**（字符级解析）。

### 6.4 桌面端工具组件不可复用

桌面端的 27 个工具组件全部绑定 DOM（`react-syntax-highlighter`、`react-markdown`、`rehype-katex`、`mermaid`）。OpenOmniBot 的 CardWidgetFactory 模式更适合我们——用 type → Card 的工厂模式，每个 Card 是纯 RN View 组件。

### 6.5 状态管理架构可以参考 OpenOmniBot 的 Mixin 拆分

当前 ChatScreen 的所有逻辑塞在一个文件里。OpenOmniBot 的 7 个 mixin 分离关注点是很好的参照：

```
ChatScreen 可以拆分为：
  chat_screen.dart           → UI 壳（导航、布局）
  chat_stream_handler.ts     → SSE 事件 → 消息状态更新
  chat_message_handler.ts    → 发送、缓存、重试
  chat_tool_renderer.ts      → 工具调用卡片渲染
  chat_lifecycle.ts          → 前后台、返回键、路由参数
```
