### 2.2 主题切换 Hook

```typescript
// src/theme/useTheme.ts
import { useColorScheme } from 'react-native';
import { MMKV } from 'react-native-mmkv';
import { lightTokens, darkTokens, ThemeTokens } from './tokens';

const storage = new MMKV();

type ThemeMode = 'light' | 'dark' | 'system';

export function useTheme(): { tokens: ThemeTokens; mode: ThemeMode; setMode: (m: ThemeMode) => void } {
  const systemScheme = useColorScheme();
  const mode = (storage.getString('theme.mode') as ThemeMode) || 'system';

  const isDark = mode === 'dark' || (mode === 'system' && systemScheme === 'dark');
  const tokens = isDark ? darkTokens : lightTokens;

  const setMode = (m: ThemeMode) => {
    storage.set('theme.mode', m);
    // 触发重渲染通过状态管理
  };

  return { tokens, mode, setMode };
}
```

## 三、SSE 客户端详细设计

### 3.1 连接状态机

```
                    ┌──────────────┐
                    │  DISCONNECTED │ ← 初始状态 / 超过最大重试
                    └──────┬───────┘
                           │ connect()
                           ▼
                    ┌──────────────┐
              ┌─────│  CONNECTING  │─────┐
              │     └──────┬───────┘     │
              │            │ onopen      │ onerror
              │            ▼             ▼
              │     ┌──────────────┐  ┌──────────────┐
              │     │  CONNECTED   │  │  RETRYING    │
              │     └──────┬───────┘  └──────┬───────┘
              │            │ onerror          │ 重试成功 → CONNECTED
              │            ▼                  │ 重试失败 → 递增延迟
              │     ┌──────────────┐          │ 超过最大次数 → DISCONNECTED
              │     │  RECONNECTING│          │
              │     └──────┬───────┘          │
              │            │ 重连成功 → CONNECTED
              │            │ 重连失败 → RETRYING
              └────────────┘
```

### 3.2 重连参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 初始延迟 | 1s | 首次断网后等待 |
| 最大延迟 | 30s | 指数退避上限 |
| 退避倍数 | 2x | 每次延迟翻倍 |
| 最大重试 | 10 次 | 超过后显示"连接断开" |
| 心跳间隔 | 15s | 检测连接存活 |

**⚠️ 移动端特殊风险与解决方案**：

| 风险 | 解决方案 |
|------|---------|
| **WiFi ↔ 4G 切换** | 使用 Android `ConnectivityManager.registerNetworkCallback` 监听网络变化，主动断开并重连（不依赖心跳超时） |
| **TCP 静默断开** | 心跳缩短到 5-10s，结合 `onerror` + 心跳超时双重检测 |
| **后台网络限制** | PoC 阶段验证 ForegroundService 方案；MVP 阶段仅支持前台 SSE，后台用本地通知兜底 |
| **OkHttp 连接池** | `react-native-sse` 的 Android polyfill 基于 OkHttp，需验证网络切换后连接池是否正确释放旧连接 |
| **初始重连延迟** | 从 1s 降到 500ms，移动端用户对延迟更敏感 |

**PoC 必须验证的场景**：
1. WiFi → 4G 切换时 SSE 自动重连
2. 飞行模式开 → 关后 SSE 恢复
3. App 后台 30s → 回前台后 SSE 重连
4. 弱网（高丢包率）下心跳保活效果

### 3.4 事件合并（Coalescing）

coalescible 事件使用 40ms 时间窗口合并：

```typescript
class EventCoalescer {
  private buffer: Map<string, SSEEvent[]> = new Map();
  private timer: ReturnType<typeof setTimeout> | null = null;
  private readonly WINDOW_MS = 40;

  push(event: SSEEvent) {
    const key = event.type;
    if (!this.buffer.has(key)) this.buffer.set(key, []);
    this.buffer.get(key)!.push(event);

    if (!this.timer) {
      this.timer = setTimeout(() => this.flush(), this.WINDOW_MS);
    }
  }

  private flush() {
    for (const [type, events] of this.buffer) {
      // 合并策略：取最后一个事件的完整数据，累加 delta 文本
      const merged = this.mergeEvents(type, events);
      this.onEvent(merged);
    }
    this.buffer.clear();
    this.timer = null;
  }

  private mergeEvents(type: string, events: SSEEvent[]): SSEEvent {
    if (type.endsWith('-delta')) {
      // delta 类型：拼接所有 text
      return {
        ...events[events.length - 1],
        data: {
          ...events[events.length - 1].data,
          text: events.map(e => e.data.text).join(''),
        },
      };
    }
    // 其他类型：取最后一个
    return events[events.length - 1];
  }
}
```

### 3.5 重连后状态恢复

```
断线 → 重连成功
  → 重新建立 SSE 连接
  → 接收 chat:system-init 恢复会话状态
  → 接收 chat:message-replay 补发断线期间的消息（SSE 自动触发）
  → 调用 GET /api/session/messages?since={lastSyncTimestamp} 增量同步（HTTP 主动拉取）
  → UI 自动更新
```

**chat:message-replay 触发条件**（源码验证）：
- SSE 连接建立时，Sidecar 自动发送当前会话的最近消息作为 replay
- replay 包含的是**当前状态**的消息（不是历史快照），所以断线期间桌面端的修改会体现在 replay 中
- replay 的消息数量有上限（通常最近 N 条），不保证完整历史

**增量同步机制**（补充 replay 不覆盖的场景）：
- replay 只包含最近消息，可能遗漏断线早期的修改
- 增量同步 API：`GET /api/session/messages?since={timestamp}&sessionId={id}`
- Sidecar 返回该时间戳之后的所有消息变更（新增 + 修改 + 删除）
- 移动端合并到本地 SQLite（INSERT OR REPLACE / DELETE）

**概要设计需补充的端点**：

| 端点 | 方法 | 用途 | 说明 |
|------|------|------|------|
| `/api/session/messages` | GET | 增量同步 | 支持 `since` 时间戳参数，返回变更消息 |

### 4.3 KaTeX 渲染设计

**资源打包方案**：将 KaTeX 和 Mermaid 的 JS/CSS 文件预打包到 App assets 中，通过 `asset://` 协议加载，不依赖 CDN。

```
myagents-companion/
├── assets/
│   ├── webview/
│   │   ├── katex.min.js          (~300KB)
│   │   ├── katex.min.css         (~20KB)
│   │   ├── mermaid.min.js        (~1.5MB)
│   │   └── webview-base.html     (模板文件)
```

**打包方式**：
- Android：放入 `android/app/src/main/assets/webview/`
- iOS：加入 Xcode project 的 Bundle Resources
- WebView 通过 `source={{ uri: 'file:///android_asset/webview/katex.html' }}` 加载（Android）或 `source={{ uri: 'bundle:///webview/katex.html' }}`（iOS）

**版本更新策略**：
- KaTeX/Mermaid 版本更新需要重新打包 App（随 App 版本发布）
- 更新频率：KaTeX/Mermaid 大版本更新时才升级（通常每年 1-2 次）
- 包体积影响：总计约 2MB，对现代手机可接受
- **备选方案**（如果包体积敏感）：首次启动时从 CDN 下载并缓存到本地文件系统，后续加载本地缓存。但增加首次启动延迟和离线首次使用的复杂度。

**优势**：零网络依赖、首次加载即时、离线可用。

// MermaidDiagram.tsx
function MermaidDiagram({ code, theme }: { code: string; theme: 'light' | 'dark' }) {
  const [state, setState] = useState<'loading' | 'ready' | 'error' | 'fallback'>('loading');
  const [svgContent, setSvgContent] = useState<string>('');
  const metricsRef = useRef<MermaidRenderMetrics>({ ... });

  const html = `
    <!DOCTYPE html>
    <html>
    <head>
      <!-- 预打包到 App assets，通过 asset:// 协议加载，不依赖 CDN -->
      <script src="asset:///mermaid.min.js"><\/script>
      <style>body { margin: 0; background: transparent; }</style>
    </head>
    <body>
      <div id="mermaid-container"></div>
      <script>
        mermaid.initialize({
          startOnLoad: false,
          theme: '${theme === 'dark' ? 'dark' : 'default'}',
          securityLevel: 'strict'
        });

        const startTime = Date.now();
        try {
          const { svg } = await mermaid.render('mermaid-graph', ${JSON.stringify(code)});
          document.getElementById('mermaid-container').innerHTML = svg;

          window.ReactNativeWebView.postMessage(JSON.stringify({
            type: 'mermaid:ready',
            svg: svg,
            renderTime: Date.now() - startTime,
            nodeCount: document.querySelectorAll('.node').length,
            svgSize: svg.length
          }));
        } catch (e) {
          window.ReactNativeWebView.postMessage(JSON.stringify({
            type: 'mermaid:error',
            error: e.message
          }));
        }
      <\/script>
    </body>
    </html>
  `;

  const handleMessage = (event: any) => {
    const data = JSON.parse(event.nativeEvent.data);
    switch (data.type) {
      case 'mermaid:ready':
        metricsRef.current = {
          nodeCount: data.nodeCount,
          svgSize: data.svgSize,
          renderTime: data.renderTime,
        };
        // 降级检查
        if (data.nodeCount > 100 || data.svgSize > 500 * 1024 || data.renderTime > 2000) {
          setState('fallback');
        } else {
          setSvgContent(data.svg);
          setState('ready');
        }
        break;
      case 'mermaid:error':
        setState('error');
        break;
    }
  };

  if (state === 'fallback' || state === 'error') {
    return <MermaidFallback code={code} error={state === 'error'} />;
  }

  return (
    <WebView
      source={{ html }}
      onMessage={handleMessage}
      style={{ height: dynamicHeight }}
      scrollEnabled={false}
    />
  );
}
```

### 6.2 缓存管理策略

```typescript
class CacheManager {
  private readonly MAX_SIZE_MB = 50;
  private readonly MAX_MESSAGES = 1000;
  private readonly MAX_DAYS = 30;

  // 写入消息（SSE 收到时异步调用）
  async insertMessage(message: CachedMessage): Promise<void> {
    await this.db.execute(
      'INSERT OR REPLACE INTO messages (id, session_id, role, content, tool_calls, created_at) VALUES (?, ?, ?, ?, ?, ?)',
      [message.id, message.sessionId, message.role, message.content, JSON.stringify(message.toolCalls), message.createdAt]
    );
    await this.evictIfNeeded();
  }

  // LRU 淘汰
  private async evictIfNeeded(): Promise<void> {
    const size = await this.getCacheSize();
    if (size <= this.MAX_SIZE_MB) return;

    // 删除最旧的会话（整个会话一起删，不拆单条消息）
    const oldestSession = await this.db.queryOne(
      'SELECT id FROM sessions ORDER BY last_message_at ASC LIMIT 1'
    );
    if (oldestSession) {
      await this.db.execute('DELETE FROM sessions WHERE id = ?', [oldestSession.id]);
      // 触发器会自动删除关联消息
    }
  }

  // 按时间淘汰（超过 30 天）
  async evictOldMessages(): Promise<void> {
    const cutoff = Date.now() - this.MAX_DAYS * 24 * 60 * 60 * 1000;
    await this.db.execute('DELETE FROM messages WHERE created_at < ?', [cutoff]);
  }

  // 查询离线消息
  async getOfflineMessages(sessionId: string, limit = 50, before?: number): Promise<CachedMessage[]> {
    let sql = 'SELECT * FROM messages WHERE session_id = ?';
    const params: any[] = [sessionId];
    if (before) {
      sql += ' AND created_at < ?';
      params.push(before);
    }
    sql += ' ORDER BY created_at DESC LIMIT ?';
    params.push(limit);
    return this.db.query(sql, params);
  }
}
```

### 6.3 缓存一致性

| 场景 | 处理方式 |
|------|---------|
| 桌面端消息修改 | 移动端不感知，重连后通过增量同步机制更新 |
| 桌面端会话删除 | 移动端保留本地缓存，显示"会话已删除"标记 |
| 移动端离线发送 | 本地队列缓存，连接恢复后自动发送 |

**增量同步机制**：

`chat:message-replay` 仅在有活跃 SSE 连接时推送，无法覆盖"离线期间桌面端修改消息"的场景。需要额外的增量同步：

```
移动端重连成功
  → 发送 GET /api/session/messages?since={lastSyncTimestamp}&sessionId={id}
  → Sidecar 返回该时间戳之后的所有消息变更
  → 移动端合并到本地 SQLite（INSERT OR REPLACE）
  → 更新 lastSyncTimestamp
```

**同步时机**：
1. SSE 重连成功后自动触发
2. 用户手动下拉刷新时触发
3. 切换会话时触发

## 七、Widget 沙箱详细设计

### 7.1 安全模型

```typescript
// WidgetRenderer.tsx
function WidgetRenderer({ widgetHtml, sessionId }: WidgetProps) {
  const [state, setState] = useState<'loading' | 'ready' | 'error' | 'fallback'>('loading');
  const failureCountRef = useRef(0);

  // 安全策略：Widget 内容由桌面端 AI 生成（非用户输入），属于"受信任内容"。
  // 但仍做基本过滤作为纵深防御：
  // 1. 使用 DOMPurify（RN 兼容版本）替代正则，防止 XSS 绕过
  // 2. WebView 的 originWhitelist=['about:blank'] 限制导航
  // 3. 不注入外部脚本，Widget 依赖的库（Chart.js/D3 等）由桌面端预渲染为 SVG 后传递
  const sanitize = (html: string): string => {
    // 使用 DOMPurify 替代正则（需安装 dompurify 包）
    // import DOMPurify from 'dompurify';
    // return DOMPurify.sanitize(html, { ALLOWED_TAGS: [...], ALLOWED_ATTR: [...] });
    return html
      .replace(/<script\b[^<]*(?:(?!<\/script>)<[^<]*)*<\/script>/gi, '')
      .replace(/\son\w+\s*=/gi, ' data-removed-')
      .replace(/javascript:/gi, 'void:');
  };

  const wrappedHtml = `
    <!DOCTYPE html>
    <html>
    <head>
      <meta name="viewport" content="width=device-width, initial-scale=1, maximum-scale=1">
      <style>
        body { margin: 0; font-family: -apple-system, system-ui, sans-serif; }
      </style>
    </head>
    <body>
      ${sanitize(widgetHtml)}
      <script>
        // 与桌面端相同的 postMessage 协议
        const RN = window.ReactNativeWebView;

        // 通知就绪
        RN.postMessage(JSON.stringify({ type: 'widget:ready' }));

        // 链接拦截
        document.addEventListener('click', (e) => {
          const link = e.target.closest('a');
          if (link) {
            e.preventDefault();
            RN.postMessage(JSON.stringify({ type: 'widget:link', url: link.href }));
          }
        });

        // 尺寸变化
        const observer = new ResizeObserver(entries => {
          const { height } = entries[0].contentRect;
          RN.postMessage(JSON.stringify({ type: 'widget:resize', height }));
        });
        observer.observe(document.body);
      <\/script>
    </body>
    </html>
  `;

  const handleMessage = (event: any) => {
    const data = JSON.parse(event.nativeEvent.data);
    switch (data.type) {
      case 'widget:ready':
        setState('ready');
        break;
      case 'widget:resize':
        setHeight(data.height);
        break;
      case 'widget:link':
        Linking.openURL(data.url);
        break;
      case 'widget:error':
        failureCountRef.current++;
        if (failureCountRef.current >= 3) {
          setState('fallback');
        } else {
          setState('error');
        }
        break;
    }
  };

  if (state === 'fallback') {
    return <WidgetFallbackCode html={widgetHtml} />;
  }

  return (
    <View>
      {state === 'loading' && <ActivityIndicator />}
      <WebView
        source={{ html: wrappedHtml }}
        onMessage={handleMessage}
        style={{ height }}
        javaScriptEnabled={true}
        allowsInlineMediaPlayback={false}
        // 安全限制
        originWhitelist={['about:blank']}
        mixedContentMode="never"
      />
    </View>
  );
}
```

### 8.1 架构方案

移动端没有中心化推送服务器，采用**本地通知**方案：

**触发链路**：

```
SSE 收到关键事件（前台时）
  → 缓存到内存队列
  → App 进入后台（AppState 监听）
  → 遍历队列，发送本地通知
  → 清空队列
```

### 8.3 通知服务实现

```typescript
// src/services/notification.ts
import PushNotification from 'react-native-push-notification';
import { AppState, AppStateStatus } from 'react-native';

class NotificationService {
  private pendingNotifications: PendingNotification[] = [];
  private isForeground = true;

  initialize() {
    // 创建通知渠道（Android）
    PushNotification.createChannel({
      channelId: 'myagents-main',
      channelName: 'MyAgents 通知',
      importance: 4, // HIGH
      vibrate: true,
    });

    // 监听 App 状态
    AppState.addEventListener('change', this.handleAppStateChange);

    // 请求权限
    PushNotification.requestPermissions();
  }

  // SSE 事件处理器调用
  onCriticalEvent(event: SSEEvent, sessionId: string) {
    const notification = this.mapEventToNotification(event, sessionId);
    if (!notification) return;

    if (this.isForeground) {
      // 前台：缓存，进入后台时再发
      this.pendingNotifications.push(notification);
    } else {
      // 后台：立即发送
      this.sendLocalNotification(notification);
    }
  }

  private handleAppStateChange = (state: AppStateStatus) => {
    const wasForeground = this.isForeground;
    this.isForeground = state === 'active';

    // 从后台回到前台时清空待发通知
    if (!wasForeground && this.isForeground) {
      this.pendingNotifications = [];
      return;
    }

    // 从前台进入后台时，发送所有待发通知
    if (wasForeground && !this.isForeground) {
      this.flushPendingNotifications();
    }
  };

  private flushPendingNotifications() {
    // 合并同类通知，避免轰炸用户
    const merged = this.mergeNotifications(this.pendingNotifications);
    for (const notification of merged) {
      this.sendLocalNotification(notification);
    }
    this.pendingNotifications = [];
  }

  private mergeNotifications(notifications: PendingNotification[]): PendingNotification[] {
    // 同一会话的通知合并为一条
    const bySession = new Map<string, PendingNotification[]>();
    for (const n of notifications) {
      const group = bySession.get(n.sessionId) || [];
      group.push(n);
      bySession.set(n.sessionId, group);
    }

    return Array.from(bySession.entries()).map(([sessionId, items]) => ({
      sessionId,
      title: items.length === 1 ? items[0].title : `MyAgents（${items.length} 条新消息）`,
      body: items.length === 1 ? items[0].body : items[items.length - 1].body,
      data: items[items.length - 1].data,
    }));
  }

  private sendLocalNotification(notification: PendingNotification) {
    PushNotification.localNotification({
      channelId: 'myagents-main',
      title: notification.title,
      message: notification.body,
      data: notification.data,
      // 点击通知跳转到对应会话
      userInfo: { sessionId: notification.sessionId },
    });
  }

  private mapEventToNotification(event: SSEEvent, sessionId: string): PendingNotification | null {
    switch (event.type) {
      case 'chat:message-complete':
        return {
          sessionId,
          title: 'AI 已回复',
          body: this.summarizeContent(event.data?.content, 50),
          data: { type: 'message', sessionId },
        };
      case 'permission:request':
        return {
          sessionId,
          title: '权限审批请求',
          body: `AI 请求使用 ${event.data?.toolName || '工具'}`,
          data: { type: 'permission', sessionId, requestId: event.data?.requestId },
        };
      case 'chat:task-notification':
        return {
          sessionId,
          title: '任务通知',
          body: event.data?.message || '任务已完成',
          data: { type: 'task', sessionId },
        };
      default:
        return null;
    }
  }

  private summarizeContent(content: string | undefined, maxLen: number): string {
    if (!content) return '点击查看';
    const text = content.replace(/[#*`\[\]()]/g, '').trim();
    return text.length > maxLen ? text.slice(0, maxLen) + '...' : text;
  }
}

interface PendingNotification {
  sessionId: string;
  title: string;
  body: string;
  data: Record<string, any>;
}

export const notificationService = new NotificationService();

import { createReadStream } from 'fs';

// GET /api/logs/export
app.get('/api/logs/export', async (c) => {
  const keyword = c.req.query('keyword') || '';
  const limit = parseInt(c.req.query('limit') || '50');
  const since = c.req.query('since'); // ISO 时间字符串

  const logDir = path.join(process.env.HOME || '', '.myagents', 'logs');
  const today = new Date().toISOString().slice(0, 10);
  const logFile = path.join(logDir, `unified-${today}.log`);

  if (!fs.existsSync(logFile)) {
    return c.json({ logs: [] });
  }

  // 流式逐行读取，避免大文件阻塞事件循环
  const results: LogEntry[] = [];
  const stream = createReadStream(logFile, { encoding: 'utf-8' });
  const rl = createInterface({ input: stream, crlfDelay: Infinity });

  for await (const line of rl) {
    if (keyword && !line.includes(keyword)) continue;
    if (since && line.slice(0, 23) < since) continue;

    results.push({
      timestamp: line.slice(0, 23),
      level: line.match(/\[(INFO|WARN|ERROR|DEBUG)\]/)?.[1] || 'INFO',
      source: line.match(/\[(REACT|RUST|NODE)\s*\]/)?.[1] || 'UNKNOWN',
      content: line.slice(35),
    });

    // 只保留最后 N 条
    if (results.length > limit) results.shift();
  }

  return c.json({ logs: results, total: results.length });
});

// src/server/middleware/mobile-auth.ts
import { jwtVerify } from 'jose';

const MOBILE_ROUTES = ['/api/session/', '/api/permission/', '/api/logs/'];
const PAIR_ROUTE = '/api/pair';

export async function mobileAuthMiddleware(c: Context, next: Next) {
  const path = c.req.path;

  // 配对端点不需要认证
  if (path === PAIR_ROUTE) return next();

  // 非移动端路由跳过
  if (!MOBILE_ROUTES.some(r => path.startsWith(r))) return next();

  const token = c.req.header('Authorization')?.replace('Bearer ', '');
  if (!token) {
    return c.json({ error: 'Missing token' }, 401);
  }

  try {
    const { payload } = await jwtVerify(token, getSecret());
    c.set('mobileSession', payload);
    return next();
  } catch (e) {
    return c.json({ error: 'Invalid or expired token' }, 401);
  }
}

// 配对端点
export async function handlePair(c: Context) {
  const { code } = await c.req.json();

  if (code !== getStoredPairCode()) {
    return c.json({ error: 'Invalid pair code' }, 403);
  }

  const token = await new SignJWT({ paired: true, device: c.req.header('User-Agent') })
    .setProtectedHeader({ alg: 'HS256' })
    .setExpirationTime('7d')
    .setIssuedAt()
    .sign(getSecret());

  return c.json({ token, expiresIn: 7 * 24 * 60 * 60 });
}

## 十一、错误处理与降级策略

### 11.1 SSE 连接错误

| 错误 | 用户看到 | 处理 |
|------|---------|------|
| 连接超时 | "连接超时" + 重试按钮 | 自动重连 + 手动重试 |
| 认证失败 | "认证失败，请重新配对" | 跳转配对页面 |
| 网络断开 | "正在重连..." + 重连动画 | 指数退避自动重连 |
| Sidecar 不在线 | "桌面端未启动" | 离线模式（显示缓存） |
| **JWT Token 过期续期失败** | "认证过期，请重新配对" | 清除该连接的 Token → 跳转配对页面 → 保留本地缓存数据 |
| **未知 host 深度链接** | "未知服务器：{host}，是否连接？" | 用户确认 → 尝试配对；用户拒绝 → 返回会话列表 |

### 11.2 渲染错误

| 错误 | 降级方式 |
|------|---------|
| WebView 初始化失败 | 显示"渲染组件加载失败" + 重试按钮 |
| postMessage 超时（5s） | 降级为代码文本 |
| WebView 内存溢出 | 关闭实例，降级为代码文本 |
| Mermaid 渲染失败 | 降级为代码文本（显示源码） |
| KaTeX 渲染失败 | 显示原始 LaTeX 文本 |
| Widget 渲染失败 | 连续 3 次后永久降级为代码文本 |

### 11.3 数据错误

| 错误 | 处理 |
|------|------|
| SQLite 写入失败 | 跳过缓存，不影响正常显示。连续失败 10 次 → 提示用户存储空间不足 |
| **单条消息缓存损坏** | 跳过该条消息，标记为 `corrupted`，下次同步时重新拉取 |
| **整个数据库损坏** | 删除数据库文件，重新创建。用户丢失离线缓存，但在线数据不受影响 |
| JSON 解析错误 | 记录日志，跳过该消息 |
| **WebView 内存溢出** | 限制同一页面最多 3 个 WebView 实例，超出时排队渲染（上一个销毁后再创建下一个） |

## 十二、深度链接设计

### 12.1 协议格式

```
myagents://s/{session_id}?h={host}

参数：
  session_id  — 会话 ID（必需）
  host        — Sidecar 地址 host:port（必需）

示例：
  myagents://s/ses_abc123?h=192.168.1.5:3210
  myagents://s/ses_abc123?h=my-tunnel.example.com:443
```

**⚠️ host 参数安全风险**：恶意构造的链接（如 `myagents://s/xxx?h=evil.com:443`）会引导 App 连接攻击者服务器。

**防护方案**：

> ⚠️ **安全修正**：原方案允许未配对 host 通过弹窗确认后连接，存在社工攻击风险（攻击者构造链接 → 用户确认 → 输入配对码 → Token 泄露）。改为**深度链接仅在已配对设备间生效**。

```
深度链接到达 → 提取 host 参数
  → 检查本地已配对列表（Keychain/Keystore）
    → 已配对 + Token 有效 → 直接连接
    → 已配对 + Token 过期 → 自动续期 → 续期成功 → 连接
                                      → 续期失败 → 提示重新配对
    → 未配对 → 拒绝连接，提示"请先在桌面端完成配对"
              → 不提供"确认连接"选项（防止社工攻击）
```

**配对流程**（独立于深度链接）：
1. 用户在桌面端开启"允许移动端连接"
2. 桌面端显示二维码（含 host + 配对码）
3. 用户用手机扫码 → App 自动配对
4. 配对成功后 host 存入白名单
5. 之后深度链接才能连接该 host

**白名单存储**：
- 存储位置：`Keychain`（iOS）/ `Keystore`（Android），与 JWT Token 同级
- 存储格式：`{ host: string, pairedAt: number, lastUsedAt: number }[]`
- 读取时机：深度链接到达时立即读取

**白名单生命周期**：
- **添加**：配对成功时自动添加
- **更新**：每次连接成功时更新 `lastUsedAt`
- **删除**：用户在设置页手动删除已配对设备，或 Token 续期失败时提示删除
- **过期**：无自动过期（配对关系持久有效，直到用户主动删除或 Token 续期失败）

**host 格式验证**：
- IP 地址：`192.168.x.x`、`10.x.x.x`、`172.16-31.x.x`（局域网 IP 直接允许）
- 域名：需要 DNS 解析验证（防止 DNS 重绑定攻击）
- 内网穿透地址：`*.ngrok.io`、`*.frp.io` 等，允许但标记为"外网连接"

### 12.2 Token 存储策略

深度链接**不携带 Token**（避免被系统日志记录）。Token 存储流程：

1. 首次配对 → JWT Token 存入 Keychain/Keystore
2. 深度链接到达 → 从本地存储读取该 host 的 Token
3. Token 过期 → 自动续期（续期失败 → 提示重新配对）

## 十四、架构设计原则（SOLID 实现指南）

> 以下为实现阶段的架构指导，确保代码可测试、可维护、可扩展。

### 14.1 单一职责（S）

**HelperService 拆分**：

§9.1 的 `HelperService` 承担了 5 项职责，建议拆分为 3 个服务：

| 服务 | 职责 | 方法 |
|------|------|------|
| `HelperSessionService` | 小助理会话生命周期 | `ensureHelperSession()`, `sendInitialMessage()`, `getHelperHistory()` |
| `HelperAdminProxy` | Admin API 调用代理 | `callAdminApi()` |
| `LogQueryService` | 日志查询 | `searchLogs()` |

**NotificationService 拆分**（P3，低优先级）：

§8.3 的 `NotificationService` 混合了事件映射、合并策略和发送逻辑，建议将 `mapEventToNotification()` 和 `mergeNotifications()` 提取为独立模块。

### 14.2 接口隔离（I）

**callAdminApi 类型安全**：

§9.1 的 `callAdminApi` 返回 `any`，建议定义端点类型映射：

```typescript
interface AdminApiEndpoints {
  '/api/provider/verify': { req: VerifyRequest; res: VerifyResult };
  '/api/mcp/list': { req: void; res: McpServer[] };
  // ...
}

async callAdminApi<T extends keyof AdminApiEndpoints>(
  endpoint: T,
  method: 'GET' | 'POST',
  body?: AdminApiEndpoints[T]['req']
): Promise<AdminApiEndpoints[T]['res']>;
```

**sidecarApi 接口分离**：

建议将 `sidecarApi` 拆分为 `ISessionApi`（会话操作）和 `IAdminApi`（管理操作），权限隔离。

**sessionStore 移动端接口**：

移动端只需要 `listSessions()` 和 `getMessages()` 子集，建议定义 `IMobileSessionStore` 接口，不暴露 `createSession()`、`deleteSession()` 等桌面端专属能力。

### 14.3 依赖倒置（D）

**存储层抽象**：

| 当前直接依赖 | 建议抽象接口 | 实现 |
|-------------|-------------|------|
| `MMKV` | `IKeyValueStorage` | `MMKVKeyValueStorage` |
| `SQLite` | `IMessageCache` | `SqliteMessageCache` |
| `sidecarApi` 全局单例 | `ISidecarApi`（构造函数注入） | `SidecarHttpApi`（由 `ConnectionManager` 创建） |
| `jose` 库 | `ITokenService` | `JoseTokenService` |

**sidecarApi 注入模式**：

```typescript
// 接口定义
interface ISidecarApi {
  get<T>(path: string): Promise<T>;
  post<T>(path: string, body?: any): Promise<T>;
}

// 通过构造函数注入（非全局 import）
class HelperSessionService {
  constructor(private api: ISidecarApi) {}
}

// 多连接管理器负责创建实例
class ConnectionManager {
  createApi(host: string, token: string): ISidecarApi {
    return new SidecarHttpApi(host, token);
  }
}
```

**优势**：单元测试可 mock、多连接场景可切换实例。

### 14.4 优先级

| 优先级 | 项目 | 说明 |
|--------|------|------|
| **高** | S-1 HelperService 拆分 | 影响小助理功能的可维护性 |
| **高** | I-1 callAdminApi 类型安全 | 防止运行时类型错误 |
| **高** | D-1/D-2 存储抽象 | 可测试性基础 |
| **高** | D-3 sidecarApi 注入 | 多连接 + 测试 |
| **中** | I-2 sidecarApi 分离 | 权限隔离 |
| **中** | I-3 sessionStore 移动端接口 | 接口精简 |
| **中** | D-4 jose 抽象 | 可替换性 |
| **低** | S-2 NotificationService 拆分 | P3 阶段再考虑 |
